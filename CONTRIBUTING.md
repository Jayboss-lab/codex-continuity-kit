# Contributing to Codex Continuity Kit

Thanks for considering a contribution. This project is a set of conventions and templates, not a codebase, so contributions should be focused, practical, and broadly useful.

---

## What we accept

- **New templates** for agent workflows not already covered
- **Improvements** to existing templates (clearer sections, better placeholders)
- **Examples** for additional project types or tech stacks
- **Documentation** fixes and clarifications
- **Translations** of template instructions (not template content itself)

## What we don't accept

- Executable code or CLI tools (this repo is templates-only)
- Vendor-specific integrations that don't generalise
- Self-promotion or product placement
- AI-generated content without human review and refinement

---

## How to contribute

1. **Open an issue first** unless it's a trivial typo fix. Use the appropriate issue template.
2. **Fork the repo**, make your changes in a branch.
3. **Follow the style** of existing templates and examples.
4. **Test your changes** by verifying the markdown renders correctly and the examples read as real projects.
5. **Open a pull request** with a clear description of what changed and why.

---

## Style guide

- **Plain markdown only.** No HTML, no special syntax beyond standard markdown.
- **Generic placeholders.** Use `[PROJECT_NAME]`, `[TECH_STACK]`, not real project names.
- **Actionable placeholders.** Each `[...]` should be a question the reader can actually answer.
- **Realistic examples.** Example projects should feel like something someone might actually build.
- **No hype.** Avoid words like "revolutionary," "AI-native," or "game-changing."

---

## Review process

All PRs require at least one review before merge. The maintainers may ask for changes or suggest alternatives.

Don't take feedback personally. This project aims for boring-useful, not flashy.

---

## Code of conduct

Be respectful, assume good intent, and focus on the problem the templates are solving.
